### Task 1: Delta read API (`/api/changes`)

**Files:** Modify `core/server.js`, `core/db.js`. Test: new `tests/delta-api.test.js`.

**Spec:**
- [ ] `core/db.js`: add `cardsSince(db, since)`, `savedSince(db, since)`, `tombstonesSince(db, since)` — rows with `updatedAt > since` (tombstones: `deletedAt > since`), mapped through the existing rowTo* functions. `since` coerced via the same `isFinite → trunc` pattern as asOf; `since` absent/0 → everything (full snapshot semantics).
- [ ] `core/server.js`: `GET /api/changes?since=<ms>` → `{ ok:true, now:<Date.now() captured BEFORE the queries>, cards:[...], saved:[...], tombstones:[...] }`. `now` is the client's next `since` watermark — capturing it before the reads means a concurrent write between capture and read is re-delivered next poll (at-least-once, never missed). Document that in a comment.
- [ ] Also `GET /api/tombstones?since=` (tombstones only — cheap poll). Both routes: read-only, no syncDirty, standard Origin guard, try/catch → the existing error middleware.
- [ ] Tests: seed cards/saved, capture `now`, mutate (upsert + delete), `GET /api/changes?since=now` returns exactly the delta incl. the tombstone; `since=0`/absent returns everything; the `now` watermark chains correctly across two polls; a concurrent-write-shaped case (write with updatedAt == now boundary) is delivered at least once (assert >= semantics — document whether boundary is > or >= and test it explicitly).
- [ ] Commit: `feat(core): delta read API — GET /api/changes?since= with tombstones (review G/Phase 4)`

### Task 2: Host allowlist + pairing-token scaffolding

**Files:** Modify `core/server.js`, `core/config.js`, `main.js` (if token generation belongs at startup). Test: extend `tests/server-security.test.js` + new `tests/pairing-auth.test.js`.

**Spec:**
- [ ] **Host allowlist (live security fix — closes the DNS-rebinding hole from the review):** a middleware before the routes rejects (403 `{ok:false,error:"forbidden host"}`) any request whose `Host` header is not `127.0.0.1:<port>`, `localhost:<port>` (any port in 3456-3465), or absent-but-loopback. Verify against real callers: the Electron renderer (Host: 127.0.0.1:port), the extension SW (fetch to http://127.0.0.1:port — Host present), curl localhost. An attacker page on evil.com DNS-rebound to 127.0.0.1 sends Host: evil.com → rejected. Must sit BEFORE the Origin guard so both run.
- [ ] **Pairing token scaffolding:** `ensurePairingToken()` in config.js — generates + persists a random 32-hex token once (crypto.randomBytes); `getPairingToken()` pure read. NOT called at startup yet (dormant — generated on first future use; but write the functions + tests now). Bearer middleware factory in server.js: `requireToken(ctx)` checks `Authorization: Bearer <token>`; WIRED so it activates only when a future config flag `lanEnabled` is true (default absent/false → middleware passes through untouched). Add `GET /api/pair-status` → `{ok:true, lan:false}` (a phone can probe capability). Document in code: flipping lanEnabled must also change the bind address — that work is NOT this release; the flag is read but the bind stays 127.0.0.1 regardless (assert that in a test so nobody flips it half-way).
- [ ] Tests: Host allowlist accept/reject matrix (incl. rebinding case + extension-style requests); token generation stable across reads; bearer middleware passthrough when lan off; enforced 401 when lan on (unit-level with a test app); bind stays loopback even with lanEnabled true in config.
- [ ] Commit: `feat(core): Host-header allowlist (DNS-rebinding fix) + dormant pairing-token auth scaffolding (review G/Phase 4)`

### Task 3: Image manifest + honest content types

**Files:** Modify `core/images.js`, `core/server.js`. Test: extend `tests/images.test.js` + endpoint test.

**Spec:**
- [ ] `core/images.js`: `imageManifest(storeDir)` → `[{id, size, type}]` using `listImageIds` + statSync + magic-byte sniffing (`sniffImageType(buf)`: JPEG ff d8 ff, PNG 89 50 4e 47, GIF, WebP RIFF....WEBP; default "image/jpeg"). Sniff reads only the first 16 bytes per file (open/read/close — not full readFile).
- [ ] `core/server.js`: `GET /api/images` → `{ok:true, images:[...manifest]}`. Fix `GET /api/img/:id` to serve the SNIFFED content type instead of hardcoded image/jpeg (review noted PNGs stored as .jpg names — the bytes are what they are; sniff the served buffer's first bytes). Renderer/extension unaffected (browsers sniff anyway; this makes the contract honest for a native client).
- [ ] Explicitly OUT of scope (document in BACKLOG via Task 6): thumbnails (needs an image-processing dependency — decide with the iOS work) and raw-binary upload (base64 PUT stays; a phone can use it).
- [ ] Tests: manifest lists seeded images with correct sizes/types (seed one real tiny PNG + one JPEG buffer); GET serves image/png for the PNG-bytes file despite its .jpg name; invalid-id and empty-store paths.
- [ ] Commit: `feat(core): image manifest endpoint + sniffed content types (review G/Phase 4)`

### Task 4: Sync robustness for future peers

**Files:** Modify `core/sync.js`, `core/merge.js` (only if needed), `core/db.js` (comment/policy). Test: extend `tests/sync-snapshot.test.js` / `tests/merge.test.js`.

**Spec:**
- [ ] **Clock-skew guard:** `readPeerSnapshots` (or the merge path) SKIPS a peer snapshot whose `meta.publishedAt` is more than 24h in the future (log via console.error, count it in the sync result as `skewSkipped`). A skewed-future peer would otherwise steamroll edits (review §1.2). Do NOT clamp item timestamps (too invasive); the snapshot-level gate + log is the Phase-4 scope. Boundary tested both sides.
- [ ] **Tombstone retention policy — decided: keep forever.** Document at `pruneTombstones` (core/db.js): intentionally never called automatically; an occasionally-offline phone peer makes any TTL unsafe; revisit only with per-peer sync cursors. Delete nothing.
- [ ] **`fp` table exclusion — decided: document.** `publishSnapshot` includes fp but merge ignores it (review §1.4): make it symmetric the CHEAP direction — stop publishing fp (placeholder fingerprints are re-derivable machine-local state), bump nothing (peers already ignore unknown files); note in the snapshot-layout comment. Verify sync-readonly/snapshot tests still pass (update the count expectations if they assert fp presence — that's a test-contract update, sanctioned).
- [ ] **Snapshot forward-compat note:** at the `schemaVersion <= SCHEMA_VERSION` gate, add the comment contract for the future phone: additive fields OK at same version; breaking changes bump the version and require app-first rollout.
- [ ] Tests: future-skewed peer skipped + counted + logged; normal peer unaffected; fp absent from new snapshots while old-format snapshots (with fp) still merge fine.
- [ ] Commit: `feat(core): clock-skew snapshot guard; fp de-published; tombstone retention policy documented (review G/Phase 4)`
